# 需求文档

## 介绍

### 产品定位

Agent Platform 是一个面向**非编码场景办公人员**的 Web 智能体对话平台。通过与在线 Agent 交流，利用 Agent 能力实现通用办公场景提效，如 Excel 数据处理、PPT 编写、文档编写等。

### 目标用户

非编码场景的企业办公人员——不具备编程能力，但希望借助 AI 智能体提升日常办公效率。

### 核心用户需求

1. **与 Agent 对话完成办公任务**：用户可以与在线 Agent 对话交流，通过 Agent 来完成通用办公场景（Excel 处理、PPT 生成、文档编写、信息检索等）。
2. **自定义并共享能力资产**：用户可以自定义知识库（仅私有，不进入市场），并可将 Agent、Skill、MCP 发布至公开市场或设为私有（MCP 仅支持配置接入，不支持开发）。
3. **灵活的模型配置**：用户可以配置自己的模型（自有 API Key）或使用平台提供的模型。

## 名词定义


| 术语             | 定义                                                                                       |
| -------------- | ---------------------------------------------------------------------------------------- |
| **Agent（智能体）** | 平台核心概念。提供除文本问答外的多种能力，包括知识库集成、Skill 使用、Remote MCP 调用、工具调用（如 Web Search）。支持用户自定义配置，随意搭配能力。 |
| **Skill（技能）**  | 遵循 Anthropic 格式的技能定义。当用户意图匹配时，Skill 自动注入到对话上下文中，增强 Agent 的处理能力。用户无需手动触发。                 |
| **Remote MCP** | 远端 Model Context Protocol 服务，用于扩展 Agent 的工具调用能力。用户仅需配置 MCP 服务地址即可接入，无需开发。                |
| **工具（Tool）**   | Agent 的基础能力集，如 Web 搜索、文件处理等。即使用户未配置任何 Skill 或 MCP，Agent 也具备基础工具能力来完成任务。                  |
| **知识库**        | 模型知识的拓展。用户可自主上传文档（PDF、Word、Excel 等），系统自动建立索引，使 Agent 能检索用户专属的领域知识。                       |


## 资产可见性模型

所有资产统一采用以下四级可见性，各资产类型的可选范围见下表：


| 可见性级别     | 含义                | Agent | Skill | MCP | 知识库 |
| --------- | ----------------- | ----- | ----- | --- | --- |
| **公开市场**  | 所有平台用户可发现、可使用     | ✔     | ✔     | ✔   | ✘   |
| **同组可编辑** | 同 W3 组织成员可查看并编辑   | ✔     | ✔     | ✔   | ✔   |
| **同组只读**  | 同 W3 组织成员可查看但不可编辑 | ✔     | ✔     | ✔   | ✔   |
| **私有**    | 仅所有者可访问           | ✔     | ✔     | ✔   | ✔   |


- 资产创建时默认可见性为**私有**。
- 知识库因包含用户业务数据，不支持发布到公开市场，最高可见性为"同组可编辑"。
- 发布到公开市场后，作者可随时将可见性调回"私有"以下架。

> 后续各需求中提到"发布/可见性选项"均指上表中该资产类型对应的可选可见性级别。

## 需求

### 需求 1：Agent 对话交互

**用户故事：** 作为办公人员，我想要与智能体对话交流，通过自然语言描述需求，让 Agent 帮我处理 Excel 数据、编写 PPT、撰写文档等日常办公任务。

#### 验收标准（EARS 格式）

1. 当用户进入对话页面时，系统应展示对话界面，包含消息区域、输入框和智能体选择器。
2. 当用户选择智能体时，系统应展示可用智能体列表，支持搜索和分类筛选。
3. 当用户在输入框输入消息并发送时，系统应将消息传递给选定的智能体并启动响应。
4. 当智能体响应时，系统应实时展示响应内容，支持流式输出显示。
5. 当智能体执行工具调用时，系统应展示工具执行状态和结果（如"正在搜索…""正在处理 Excel…"）。
6. 当用户上传文件（Excel、PPT、Word、PDF、图片等）时，系统应支持文件解析并传递给 Agent 处理。上传文件受需求 12 文件安全策略约束（类型、大小限制）。
7. 当 Agent 生成文件（如处理后的 Excel、生成的 PPT）时，系统应提供文件预览和下载功能。预览应至少支持 PDF、Excel（表格渲染）、图片的在线查看；其他格式提供直接下载。
8. 对话中上传和生成的文件应保留至少 **30 天**（可由平台管理员调整），到期后系统自动清理并在文件位置显示"文件已过期"提示。
9. 文件下载链接应绑定当前用户会话，未登录或非对话参与者访问时返回 403；下载链接自生成起 **24 小时**内有效，过期后用户可在对话中重新获取。
10. 当用户点击消息的"重新生成"按钮时，系统应重新请求智能体生成响应。
11. 当用户点击消息的"复制"按钮时，系统应复制消息内容到剪贴板。
12. 当用户点击"清空对话"按钮时，系统应弹出确认后清空当前对话历史。
13. 如果对话超过当前智能体配置的最大步骤数（见需求 2 第 2 条），则系统应提示用户并停止响应，同时提供"继续执行"和"调整步骤上限"的操作入口。
14. 当用户切换智能体时，系统应在 UI 上保留当前对话历史供用户回顾，但**不**将历史消息作为上下文传递给新 Agent（新 Agent 从空白上下文开始）。切换时应在对话流中插入可见的分隔标记（如"— 已切换到 {新 Agent} —"），明确区分前后上下文边界。

### 需求 2：智能体管理

**用户故事：** 作为办公人员，我想要创建和管理智能体，根据办公需求定制专属的 AI 助手（如"Excel 数据分析助手""会议纪要生成器"等）。

#### 验收标准（EARS 格式）

1. 当用户进入"创建智能体"页面时，系统应展示智能体创建的引导流程。
2. 当用户点击"创建智能体"按钮时，系统应提供智能体基本信息配置表单，包括名称、描述、头像、系统提示词（Prompt）和最大步骤数。最大步骤数为每轮对话中 Agent 可执行的工具调用步数上限，按单个 Agent 维度配置，默认值 10，用户可调整范围为 1–50。
3. 当用户配置智能体模型时，系统应允许用户从平台提供的模型列表中选择，或配置自有模型 API Key（见需求 9）。
4. 当用户配置智能体工具时，系统应提供可视化的工具选择界面（勾选启用），支持按分组浏览（内置工具、MCP 工具、知识库检索）。
5. 当用户配置智能体的 Skill 时，系统应提供已有 Skill 的选择列表，支持勾选绑定。
6. 当用户完成智能体配置时，系统应保存智能体并提供调试功能进行测试验证。
7. 当用户在智能体列表页面时，系统应展示用户拥有的所有智能体卡片，包括名称、描述、状态和操作按钮。
8. 当用户点击智能体卡片的"编辑"按钮时，系统应跳转到智能体配置界面进行修改。
9. 当用户点击智能体卡片的"复制"按钮时，系统应创建智能体副本并添加到列表。
10. 当用户点击智能体卡片的"删除"按钮时，系统应弹出确认对话框，确认后删除智能体。
11. 当用户点击智能体卡片的"导出"按钮时，系统应生成配置文件供下载。
12. 当用户导入智能体配置文件时，系统应验证文件格式并创建智能体副本。

### 需求 3：智能体市场发布与发现

**用户故事：** 作为办公人员，我想要将我的智能体发布到公开市场或设为私有，也想从市场发现他人分享的实用智能体。

#### 验收标准（EARS 格式）

1. 当用户在智能体配置界面完成调试并验证智能体可用时，系统应提供"发布"按钮。
2. 当用户点击"发布"按钮时，系统应弹出发布确认对话框，包含可见性选项（公开市场 / 同组可编辑 / 同组只读 / 私有，参见"资产可见性模型"）。
3. 当用户选择可见性并确认时，系统应按所选级别发布智能体，状态变更为"已发布"。
4. 当用户在智能体市场页面时，系统应展示所有公开发布的智能体列表，支持分类浏览和搜索。
5. 当用户点击市场智能体卡片时，系统应展示智能体详情，包括名称、描述、作者、能力列表和使用统计。
6. 当用户点击市场智能体的"使用"按钮时，系统应创建对话会话并启动该智能体。
7. 当用户点击市场智能体的"收藏"按钮时，系统应将智能体添加到用户收藏列表。
8. 如果智能体已被发布，则系统应支持版本管理，允许作者查看历史版本和回滚。
9. 当已发布智能体有更新时，系统应提示作者是否发布新版本。

### 需求 4：Skill 管理

**用户故事：** 作为办公人员，我想要创建和管理 Skill，定义可复用的办公技能（如"邮件润色""数据图表生成"），当对话中涉及相关意图时自动生效。

#### 验收标准（EARS 格式）

1. 当用户进入 Skill 管理页面时，系统应展示用户拥有的所有 Skill 卡片列表。
2. 当用户点击"创建 Skill"按钮时，系统应提供 Skill 创建表单，包括名称、描述和触发条件。
3. 当用户编写 Skill 内容时，系统应提供模板化编辑器（支持 Anthropic 格式），降低非编码用户的使用门槛。
4. 当用户完成 Skill 创建时，系统应保存 Skill 并验证格式正确性。
5. 当用户点击 Skill 卡片的"编辑"按钮时，系统应打开 Skill 编辑界面。
6. 当用户点击 Skill 卡片的"删除"按钮时，系统应弹出确认对话框后删除。
7. 当用户点击 Skill 卡片的"导出"按钮时，系统应生成 Skill 配置文件供下载。

### 需求 5：Skill 市场发布与发现

**用户故事：** 作为办公人员，我想要将 Skill 发布到市场或设为私有，也想从市场发现他人分享的实用 Skill。

#### 验收标准（EARS 格式）

1. 当 Skill 市场页面加载时，系统应展示所有公开发布的 Skill 列表，支持分类和搜索。
2. 当用户点击 Skill 市场卡片时，系统应展示 Skill 详细信息，包括描述、使用场景和示例。
3. 当用户点击 Skill 市场卡片的"导入"按钮时，系统应将 Skill 复制到用户资产列表。
4. 当用户在 Skill 管理页面点击"发布"按钮时，系统应提供可见性选项（参见"资产可见性模型"）。
5. 如果 Skill 已发布到市场，则系统应展示使用次数和评分统计。
6. 当用户对已使用的 Skill 进行评价时，系统应记录评分并更新统计。

### 需求 6：MCP 管理

**用户故事：** 作为办公人员，我想要配置 Remote MCP 服务来扩展 Agent 的工具能力（如接入企业内部 API、第三方办公服务），仅需填写服务地址即可，无需开发。

#### 验收标准（EARS 格式）

1. 当用户进入 MCP 管理页面时，系统应展示用户配置的所有 MCP 服务列表。
2. 当用户点击"添加 MCP"按钮时，系统应提供简洁的配置表单（服务名称、服务地址 URL、协议类型选择）。
3. 当用户填写 MCP 服务地址时，系统应支持 SSE 和 Streamable HTTP 两种协议。
4. 当用户完成 MCP 配置时，系统应自动验证连接有效性并保存。
5. 当用户点击 MCP 卡片的"测试连接"按钮时，系统应测试 MCP 连接状态并返回结果（如"连接成功，发现 N 个工具"）。
6. 当用户点击 MCP 卡片的"删除"按钮时，系统应弹出确认后删除 MCP 配置。
7. 当用户点击 MCP 卡片的"启用/禁用"切换时，系统应更新 MCP 服务的可用状态。
8. 如果 MCP 服务连接失败，则系统应展示错误信息并提供重连选项。

### 需求 7：MCP 市场发布与发现

**用户故事：** 作为办公人员，我想要从 MCP 市场发现可用的 MCP 服务，一键接入来扩展 Agent 能力，也可将自己配置的 MCP 分享到市场或设为私有。

#### 验收标准（EARS 格式）

**发现与接入**

1. 当 MCP 市场页面加载时，系统应展示所有可用的公开 MCP 服务列表。
2. 当用户在 MCP 市场搜索框输入关键词时，系统应返回匹配的 MCP 结果。
3. 当用户点击 MCP 市场卡片时，系统应展示 MCP 详细信息，包括服务描述、提供的工具列表和使用说明。
4. 当用户点击 MCP 市场卡片的"接入"按钮时，系统应将 MCP 元信息（名称、URL、协议类型、工具列表）复制到用户资产，并引导用户自行填写认证信息。
5. 如果 MCP 服务不需要认证，则系统应跳过认证填写步骤，直接完成接入。

**发布**

1. 当用户在 MCP 管理页面点击"发布"按钮时，系统应提供可见性选项（参见"资产可见性模型"）。
2. 当用户将 MCP 发布到公开市场时，系统应**仅发布元信息**（服务名称、描述、URL、协议类型、工具列表），**不得包含发布者的认证凭据**（如 Header Token、API Key）。
3. 当 MCP 服务有更新时，系统应提示用户更新版本。

### 需求 8：知识库管理

**用户故事：** 作为办公人员，我想要上传自己的文档资料建立知识库，让 Agent 在对话时能检索这些资料，回答更准确、更贴合业务。

#### 验收标准（EARS 格式）

1. 当用户进入知识库管理页面时，系统应展示用户拥有的所有知识库列表。
2. 当用户点击"创建知识库"按钮时，系统应提供知识库创建表单，包括名称和描述。
3. 当用户上传文档到知识库时，系统应支持常见办公格式（PDF、Word、Excel、PPT、TXT、Markdown），自动解析内容并建立索引。
4. 当用户点击知识库卡片的"管理文档"按钮时，系统应展示知识库中的所有文档列表，显示文件名、大小、索引状态。
5. 当用户删除知识库中的文档时，系统应更新索引并移除文档。
6. 当用户点击知识库卡片的"测试检索"按钮时，系统应提供检索测试界面，输入查询返回匹配结果。
7. 当用户配置智能体时，系统应允许选择知识库检索工具并配置目标知识库。
8. 如果知识库索引失败，则系统应展示错误信息并支持重试。

### 需求 9：模型配置与管理

**用户故事：** 作为办公人员，我想要灵活选择 AI 模型——可以使用平台提供的模型快速上手，也可以在独立页面集中管理自己的模型 API Key，在智能体间复用。

#### 验收标准（EARS 格式）

**在智能体中选择模型**

1. 当用户创建或编辑智能体时，系统应提供模型选择下拉列表，同时展示平台内置模型和用户已添加的自定义模型。
2. 当用户选择平台内置模型时，系统应直接绑定该模型到智能体，无需额外配置。

**自定义模型管理（独立页面）**

1. 当用户进入"模型管理"页面时，系统应展示用户已添加的所有自定义模型列表，包括模型名称、API 地址、连接状态和绑定的智能体数量。
2. 当用户点击"添加自定义模型"按钮时，系统应提供配置表单，包括模型名称、API 地址和 API Key。
3. 当用户保存自定义模型配置时，系统应验证 API 连通性，验证通过后保存配置。
4. 当用户编辑已有自定义模型的 API Key 或 API 地址时，系统应重新验证连通性，验证通过后更新配置，**所有绑定该模型的智能体立即生效**，无需逐个修改。
5. 当用户删除自定义模型时，如果该模型仍被智能体引用，系统应弹出确认对话框列出受影响的智能体，确认后将这些智能体的模型重置为平台默认模型。
6. 如果用户的自定义模型 API 连接失败，则系统应展示错误信息并提示检查配置。

### 需求 10：W3 OAuth 认证集成与权限

**用户故事：** 作为办公人员，我想要通过公司 W3 认证系统登录平台，无缝访问并管理我的资产。

#### 验收标准（EARS 格式）

1. 当用户首次访问平台时，系统应自动检测 W3 登录状态，未登录则跳转到 W3 OAuth 授权页面。
2. 当用户在 W3 OAuth 页面完成授权时，系统应接收 OAuth 回调，获取用户身份信息（用户 ID、姓名、邮箱、组织等）。
3. 当用户通过 W3 OAuth 成功登录时，系统应生成平台会话令牌并加载用户资产。
4. 当用户会话过期时，系统应自动刷新令牌或重新引导用户通过 W3 OAuth 登录。
5. 当用户创建资产时，系统应自动设置资产所有者为当前 W3 用户。
6. 当用户设置资产可见性时，系统应按"资产可见性模型"提供该资产类型支持的可见性级别选项。
7. 如果用户尝试访问无权限的资产，则系统应拒绝访问并提示权限不足。
8. 当用户退出登录时，系统应清除平台会话并跳转到 W3 登出页面。
9. 当管理员查看用户管理页面时，系统应展示从 W3 同步的用户列表和权限配置选项。
10. 当管理员修改用户权限时，系统应即时更新权限配置。

### 需求 11：资产管理统一入口

**用户故事：** 作为办公人员，我想要有一个清晰的统一入口管理我的所有资产，快速找到和组织我的智能体、Skill、MCP 和知识库。

#### 验收标准（EARS 格式）

1. 当侧边栏导航加载时，系统应展示清晰的资产分类：智能体、Skill、MCP、知识库。
2. 当用户点击导航菜单项时，系统应切换到对应的资产管理页面。
3. 当用户点击"创建"入口时，系统应展示创建引导页面，包含智能体、Skill、MCP、知识库的创建选项，以可视化卡片引导用户选择。
4. 当用户点击创建引导卡片的"开始创建"按钮时，系统应跳转到对应的创建表单页面。
5. 如果用户没有创建权限，则系统应隐藏创建入口并提示权限不足。

### 需求 12：数据安全与隐私

**用户故事：** 作为企业办公人员，我需要平台对我的敏感信息（模型 API Key、MCP 认证凭据、上传的业务文档）提供安全保障，防止泄露和越权访问。

#### 验收标准（EARS 格式）

**凭据安全**

1. 当用户提交自定义模型 API Key 时，系统应对 API Key 进行加密存储，界面上仅展示脱敏后的部分字符（如 `sk-****abcd`）。
2. 当用户配置 MCP 认证信息（如 Header Token）时，系统应对认证凭据进行加密存储，不以明文形式出现在日志、接口响应或前端代码中。
3. 当用户删除自定义模型或 MCP 配置时，系统应同步销毁关联的加密凭据，不做保留。

**文件安全**

1. 当用户上传文件到对话或知识库时，系统应限制单文件大小上限（建议 50MB）和允许的文件类型（PDF、Word、Excel、PPT、TXT、Markdown、常见图片格式），拒绝超限或不合法的文件并提示原因。
2. 当用户上传文件时，系统应对文件进行安全扫描（如类型校验、恶意内容检测），拒绝不安全的文件。
3. 当文件存储在服务端时，系统应按用户/组织维度进行隔离存储，不同用户之间不可互相访问未授权的文件。

**权限隔离**

1. 当用户访问任何资产（Agent、Skill、MCP、知识库）时，系统应进行权限校验：私有资产仅所有者可访问，同组资产按"同组可编辑/同组只读"规则控制。
2. 当用户通过 API 或直接 URL 尝试访问他人私有资产时，系统应返回 403 并记录异常访问日志。
3. 如果用户的 Agent 绑定了其他用户的公开 MCP 服务，则 Agent 调用该 MCP 时不应暴露 MCP 所有者的认证凭据给调用方。

**审计与合规**

1. 当发生敏感操作（创建/删除 API Key、发布/下架资产、管理员修改权限）时，系统应记录操作审计日志，包含操作人、操作时间和操作内容。
