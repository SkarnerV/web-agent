# Playwright MCP E2E 测试计划

## 1. 测试概述

### 1.1 测试目标

对本 Agent 平台的前端应用进行全面的端到端功能测试，验证以下核心质量属性：

- **功能正确性**：所有页面可正常加载，CRUD 操作流程完整可用
- **API 集成**：前端与后端 REST API 的数据交互正确（请求/响应/错误处理）
- **用户体验**：关键用户流程（创建 Agent、发送聊天消息、导入市场资产等）可用且无阻塞性缺陷
- **UI 一致性**：页面布局、组件渲染、状态展示（loading/empty/error）符合设计预期
- **SSE 流式通信**：聊天消息的流式传输正常（token 渲染、工具调用、引用展示）

### 1.2 测试范围

| 模块 | 页面数 | 测试用例数 | 优先级分布 |

### 2.3 测试数据管理

- **命名规范**：所有测试数据使用中文，便于识别和调试
  - Agent 名称：`测试Agent_自动化_001`
  - Skill 名称：`测试Skill_文本处理_001`
  - 知识库名称：`测试知识库_产品文档_001`
- **数据隔离**：每个测试用例使用唯一的时间戳后缀，避免并行冲突
- **清理策略**：每个测试套件在 `afterAll` 中清理创建的测试数据（通过 API 删除）

## 6. 风险与注意事项

1. **SSE 流式测试不稳定**：网络波动可能导致 token 到达顺序和数量不一致，断言应使用 `≥` 而非精确数量匹配
2. **测试数据污染**：每个用例使用唯一时间戳后缀命名，测试结束后通过 API 清理，避免影响后续运行
3. **后端依赖**：API 集成测试依赖后端数据状态，某些用例（如空状态测试）可能需要专门的测试环境
4. **MCP 连接测试**：测试 MCP 连接需要可访问的外部 MCP 服务器，建议搭建本地 Mock MCP 服务器用于测试
5. **文件上传测试**：知识库文档上传测试需要准备标准测试文件，建议放在 `e2e/fixtures/files/` 目录下
6. **时间角色**：聊天会话按"今天/昨天/更早"分组，跨天运行可能导致分组变化，断言应使用正则匹配
